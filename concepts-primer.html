Document Chunking — The Most Impactful Decision

Chunking is where you decide what unit of information the retriever can see. Too small, and the chunk lacks context. Too large, and irrelevant content dilutes the signal. The right strategy depends on your document type, and the only way to know is to experiment. P5 requires implementing at least 3 strategies so you can measure which works best on your specific PDFs.

First Principle

A retrieval system can only return what it was given. If a critical answer spans two chunks and neither chunk alone is complete, the LLM will generate an incomplete answer — no matter how good the model is. Chunking defines the atoms of your search space. Get the atoms wrong, and everything downstream suffers. This is the "garbage in, garbage out" principle applied at the granularity level.

Strategy 1: Fixed-Size Chunking

How It Works

Split text into chunks of exactly N tokens (or characters), with an overlap of M tokens between consecutive chunks. Simple, predictable, and fast. The overlap ensures that information at chunk boundaries isn't lost.

Token vs Character counting: Use tiktoken for accurate token counts. Characters are unreliable because tokenization is non-uniform — "extraordinary" is 1 word but often 3+ tokens. A 512-character chunk might be 120 tokens or 180 tokens depending on content.

# WHY tiktoken: LLMs process tokens, not characters.
# A chunk that's "512 characters" might be 100 tokens (simple English)
# or 200 tokens (technical jargon with many sub-word pieces).
# Token-based chunking gives consistent semantic density.

import tiktoken

# WHY cl100k_base: This is the tokenizer for GPT-4/GPT-3.5.
# If you embed with OpenAI models, use the same tokenizer for consistency.
# For sentence-transformers, the model has its own tokenizer,
# but cl100k_base is a reasonable proxy for "semantic token count."
enc = tiktoken.get_encoding("cl100k_base")

def fixed_size_chunk(text: str, chunk_size: int = 512, overlap: int = 128) -> list[str]:
    """Split text into fixed-size token chunks with overlap."""
    tokens = enc.encode(text)
    chunks = []
    # WHY stride = chunk_size - overlap: This creates the sliding window.
    # With chunk_size=512 and overlap=128, stride=384.
    # Each new chunk starts 384 tokens after the previous one,
    # sharing 128 tokens with its predecessor.
    stride = chunk_size - overlap
    for i in range(0, len(tokens), stride):
        chunk_tokens = tokens[i : i + chunk_size]
        chunks.append(enc.decode(chunk_tokens))
    return chunks

Strategy 2: Recursive Chunking

How It Works

Split on a hierarchy of separators: first try \n\n (paragraph breaks), then \n (line breaks), then . (sentences), then space. This preserves document structure — paragraphs stay together, sections stay together. Only falls to smaller separators when a chunk exceeds the target size.

Why it's often the best default: Most documents have natural paragraph structure. Splitting on paragraph boundaries produces chunks where each chunk is a coherent thought. Fixed-size chunking can split mid-sentence.

# WHY a hierarchy of separators: Documents have structure.
# A research paper has sections → paragraphs → sentences.
# Try the biggest separator first (section break), only fall to
# smaller ones (sentence break) if the section exceeds chunk_size.

SEPARATORS = ["\n\n", "\n", ". ", " "]

def recursive_chunk(text: str, chunk_size: int = 512, separators: list = None) -> list[str]:
    """Recursively split text using a hierarchy of separators."""
    separators = separators or SEPARATORS
    if not separators:
        return [text]  # Base case: no separators left, return as-is

    sep = separators[0]
    splits = text.split(sep)
    chunks, current = [], ""

    for piece in splits:
        # WHY check before appending: If adding this piece exceeds
        # chunk_size, flush current chunk and start a new one.
        if len(enc.encode(current + sep + piece)) > chunk_size:
            if current:
                chunks.append(current.strip())
            # If the piece itself exceeds chunk_size, recurse with
            # the next separator in the hierarchy
            if len(enc.encode(piece)) > chunk_size:
                chunks.extend(recursive_chunk(piece, chunk_size, separators[1:]))
                current = ""
            else:
                current = piece
        else:
            current = current + sep + piece if current else piece

    if current:
        chunks.append(current.strip())
    return chunks

Strategy 3: Semantic Chunking

How It Works

Embed individual sentences, then measure cosine similarity between consecutive sentences. When similarity drops below a threshold (a "breakpoint"), you've found a topic boundary. Chunks are formed by grouping sentences between breakpoints. This is computationally expensive — you must embed every sentence — but produces the most semantically coherent chunks.

Key insight: This is the only strategy that "reads" the content. Fixed-size and recursive are purely structural — they don't know if two sentences are about the same topic. Semantic chunking does.

# WHY embed every sentence: We need to detect TOPIC SHIFTS.
# Two consecutive sentences about "neural network training" will have
# high cosine similarity (~0.8+). When the topic shifts to "deployment costs",
# similarity drops (~0.3). That drop is our chunk boundary.

import numpy as np
from sentence_transformers import SentenceTransformer

def semantic_chunk(text: str, model: SentenceTransformer, threshold: float = 0.5) -> list[str]:
    """Split text at semantic breakpoints detected via embedding similarity."""
    sentences = text.split(". ")  # Naive sentence split — use nltk/spacy for production
    embeddings = model.encode(sentences, normalize_embeddings=True)

    # WHY consecutive cosine similarity: We're looking for topic TRANSITIONS.
    # We don't care about absolute similarity — we care about DROPS.
    chunks, current_chunk = [], [sentences[0]]
    for i in range(1, len(sentences)):
        # WHY dot product on normalized vectors = cosine similarity
        # (You proved this in P2/P3 with FAISS IndexFlatIP)
        sim = np.dot(embeddings[i - 1], embeddings[i])
        if sim < threshold:
            chunks.append(". ".join(current_chunk))
            current_chunk = [sentences[i]]
        else:
            current_chunk.append(sentences[i])

    if current_chunk:
        chunks.append(". ".join(current_chunk))
    return chunks

Strategy 4: Sliding Window

How It Works

Like fixed-size, but with very high overlap (50%+). A 512-token chunk with 256-token overlap means every piece of text appears in at least two chunks. This maximizes recall at the cost of index size and search speed — more chunks means more candidates to search through.

When to use: When missing a relevant passage is more costly than returning extra results. Legal document search, compliance checks, safety-critical Q&A.

Choosing the Right Strategy

Document TypeBest StrategyChunk SizeOverlapWhy
Research papers Recursive 512–1024 50–100 Preserves section structure (Abstract, Methods, Results)
Legal contracts Semantic 256–512 64–128 Clauses are self-contained — semantic boundaries align with legal boundaries
Technical manuals Fixed-size 512 128 Dense factual information, high overlap needed to avoid boundary losses
Mixed / Unknown Sliding window 512 256 Maximizes recall when you can't predict where answers live
Java/TypeScript Parallel

Chunking strategies map to the Strategy Pattern. In Java: interface Chunker { List<Chunk> chunk(Document doc); } with FixedSizeChunker, RecursiveChunker, SemanticChunker implementations. In Python, the same pattern uses Abstract Base Classes (abc.ABC) with @abstractmethod. The config file selects which strategy to use at runtime — same as dependency injection, just more explicit.

Pitfall

Don't split mid-sentence. Even fixed-size chunking should snap to sentence boundaries. A chunk ending with "The primary cause of the failure was" and the next starting with "corrosion in the left engine mount" means neither chunk contains the full answer. Use sentence-boundary-aware splitting — find the last sentence end before the token limit.

Embedding Models — Quality vs Speed Trade-off

You used embedding models in P2 and P3, but always with a single model per experiment. P5 requires testing at least 2 and measuring the quality vs speed trade-off. The key insight: higher dimensions capture more semantic nuance but cost more compute and storage.

Model Comparison

ModelDimsSpeedQualityBest For
all-MiniLM-L6-v2 384 Fast Good Production with large corpora — where you need speed over precision
all-mpnet-base-v2 768 Medium Better Quality-critical applications where you can afford slower indexing
multi-qa-MiniLM-L6-cos-v1 384 Fast Good (QA-optimized) Question-answering specifically — trained on QA pairs, may outperform general models on your task
First Principle

Embedding is lossy compression. You're taking a sequence of tokens (variable length) and squeezing it into a fixed-dimension vector. Higher dimensions = more capacity to preserve nuance. But there's diminishing returns — going from 384d to 768d doubles storage and compute but doesn't double quality. This is the bias-variance trade-off applied to representation learning: more dimensions reduce bias (underfitting) but increase computational cost.

# WHY normalize_embeddings=True: Without normalization, vectors have
# varying magnitudes. Cosine similarity = dot_product / (|a| * |b|).
# If you pre-normalize (|v| = 1), then cosine_sim = dot_product.
# This is why FAISS IndexFlatIP works for cosine similarity after normalization.
# You proved this equivalence in P2 and P3.

from sentence_transformers import SentenceTransformer

# WHY two separate model instances: Each model has different tokenization
# and different output dimensions. You can't mix embeddings from different
# models in the same FAISS index — the geometry is incompatible.
fast_model = SentenceTransformer("all-MiniLM-L6-v2")      # 384d
quality_model = SentenceTransformer("all-mpnet-base-v2")   # 768d

# Encode with normalization — mandatory for cosine similarity via FAISS IP
embeddings = fast_model.encode(
    texts,
    normalize_embeddings=True,   # L2-normalize each vector to unit length
    show_progress_bar=True,      # Useful for large corpora
    batch_size=32                 # WHY 32: Fits in 8GB M2 RAM comfortably
)
M2 Memory Warning

On your 8GB MacBook Air M2, loading two SentenceTransformer models simultaneously may cause memory pressure. Load one model, embed, save the index, release the model (del model), then load the next. Monitor with psutil.virtual_memory(). The all-mpnet-base-v2 model is ~420MB — two models plus FAISS indices could push past available RAM.

Key Insight

You can't mix embeddings from different models in the same FAISS index. A 384d MiniLM vector and a 768d mpnet vector live in completely different geometric spaces. Build a separate index for each model. This also means your experiment grid must track which index to load for each configuration.

FAISS & Vector Indexing — Beyond P2

You used FAISS in P2 and P3, but P5 adds persistence (save/load indices to disk), metadata association (map vector indices back to chunk objects), and multiple index management (one index per embedding model). The core search mechanics are the same — IndexFlatIP for exact cosine similarity on normalized vectors.

Persistence Pattern

import faiss
import json
from pathlib import Path

class FAISSVectorStore:
    """FAISS index with chunk metadata persistence."""

    def save(self, path: Path):
        """Save index + metadata to disk."""
        # WHY save both: FAISS only stores vectors — it has NO knowledge of
        # what those vectors represent. The metadata mapping (index position → chunk)
        # must be saved separately. If you lose the metadata, the vectors are useless.
        faiss.write_index(self.index, str(path / "index.faiss"))
        with open(path / "chunks.json", "w") as f:
            json.dump([c.model_dump() for c in self.chunks], f)

    def load(self, path: Path):
        """Load index + metadata from disk."""
        # WHY faiss.read_index: Rebuilding a FAISS index from scratch means
        # re-embedding every chunk (~minutes for 1000 chunks). Loading a saved
        # index is ~milliseconds. This is the "don't rebuild every time" principle.
        self.index = faiss.read_index(str(path / "index.faiss"))
        with open(path / "chunks.json") as f:
            self.chunks = [Chunk(**c) for c in json.load(f)]
Java/TypeScript Parallel

Think of FAISS persistence like serializing a HashMap<Integer, Chunk> plus a spatial index to disk. In Java, you'd use ObjectOutputStream for the map and a custom serializer for the index. The pattern is the same: the index gives you fast lookup by vector similarity, but the metadata (what each position means) must be stored separately. This is why databases separate indices from data — same principle.

BM25 & Hybrid Retrieval — Score Fusion

Dense retrieval (embedding similarity) is great for semantic matching — "How do I reset my password?" matches "Password recovery instructions." But it can miss exact keyword matches that BM25 catches: searching for "error code E-4012" won't match a semantically similar passage unless the exact string appears. Hybrid retrieval combines both to get the best of both worlds.

First Principle

Dense retrieval and BM25 fail in complementary ways. Dense fails on rare/technical terms (low embedding coverage). BM25 fails on synonyms and paraphrases (no semantic understanding). Combining them is a ensemble method — the same principle behind Random Forests and model ensembles. Two weak signals combined through a learned or tuned weighting often outperform either signal alone.

BM25 — The Sparse Baseline

What BM25 Actually Computes

BM25 is a TF-IDF variant with two improvements: (1) term frequency saturation (additional occurrences of a word have diminishing impact), and (2) document length normalization (longer documents don't get unfair advantage). For each query term, it scores each document based on how rare the term is (IDF) and how frequently it appears in that document (TF), adjusted for document length.

Key intuition: A rare term that appears frequently in a document is a strong match signal. A common term ("the", "is") that appears everywhere is not.

# WHY rank_bm25: The standard Python BM25 library. Lightweight,
# no dependencies, ~5 lines to use. Think of it as the "grep" of
# information retrieval — simple but surprisingly effective.

from rank_bm25 import BM25Okapi

# WHY tokenize as lists of words: BM25 operates on TERM FREQUENCY.
# It needs the raw word lists, not embeddings. Each document is a
# list of tokens: ["the", "cat", "sat", "on", "the", "mat"]
tokenized_corpus = [chunk.content.lower().split() for chunk in chunks]
bm25 = BM25Okapi(tokenized_corpus)

# Score a query against all documents
query_tokens = "how to reset password".lower().split()
scores = bm25.get_scores(query_tokens)  # Array of scores, one per chunk

The Score Fusion Problem

Why Naive Combination Fails

Dense scores are cosine similarity: range [0, 1]. BM25 scores are TF-IDF based: range [0, ∞). If you simply add them, BM25 dominates because its scores can be 10x, 100x larger. You must normalize before combining.

# WHY min-max normalization: Bring BM25 scores to [0, 1] range
# so they're comparable with cosine similarity scores.
# Without this, hybrid retrieval = BM25 with decorative dense scores.

def normalize_scores(scores: list[float]) -> list[float]:
    min_s, max_s = min(scores), max(scores)
    if max_s == min_s:
        return [0.0] * len(scores)  # All scores identical → all zero
    return [(s - min_s) / (max_s - min_s) for s in scores]

# WHY alpha parameter: Controls the dense vs sparse weight.
# alpha = 1.0 → pure dense.  alpha = 0.0 → pure BM25.
# alpha = 0.7 → 70% dense + 30% BM25 (typical starting point).
# The spec says to test α ∈ {0.3, 0.5, 0.7} — this is a hyperparameter.

def hybrid_score(dense_scores, bm25_scores, alpha=0.7):
    norm_bm25 = normalize_scores(bm25_scores)
    return [
        alpha * d + (1 - alpha) * b
        for d, b in zip(dense_scores, norm_bm25)
    ]
Key Insight from P2

In P2, you found that reranking improved Recall@5 from baseline to 0.747 — but that was dense-only retrieval. P5's hybrid retrieval may achieve similar or better recall without reranking, because BM25 catches the exact-match cases that dense retrieval misses. Your experiment grid should include: dense-only, BM25-only, and hybrid with α ∈ {0.3, 0.5, 0.7} to prove or disprove this hypothesis.

🔍 Analogy — Two Librarians

Dense retrieval is a librarian who understands meaning: "I need information about climate policy" and they bring you books about environmental regulation, carbon pricing, and sustainability targets — even though none of those words were in your query.

BM25 is a librarian who's obsessive about exact words: they'll find every document containing "climate" and "policy" — including the one with the exact quote you need, but also a bunch of irrelevant documents that happen to contain those words.

Hybrid retrieval is hiring both librarians and merging their recommendations. The semantic librarian finds conceptually relevant documents. The keyword librarian catches exact matches. Together, they cover each other's blind spots.

Reranking — Cross-Encoders vs Bi-Encoders

In P2, you used Cohere's reranking API to improve retrieval results. P5 asks you to understand why reranking works — and the fundamental architectural difference between the bi-encoder you use for embedding and the cross-encoder used for reranking.

First Principle

There's a fundamental trade-off between speed and interaction depth. A bi-encoder embeds query and document independently — fast (O(1) per query) but no token-level interaction between query and document. A cross-encoder processes query+document together through all transformer layers — slow (O(n) per candidate) but captures fine-grained relationships. This is the same retrieve-then-rerank pattern used in web search: Google first retrieves 1000 candidates with a fast model, then reranks the top 100 with a slow, accurate model.

PropertyBi-Encoder (Retrieval)Cross-Encoder (Reranking)
Input Query and document encoded separately Query and document encoded together as one sequence
Output Two vectors → cosine similarity Single relevance score (0-1)
Speed Fast — precompute doc vectors once, search in ms Slow — must re-encode every (query, doc) pair at query time
Accuracy Good — captures semantic similarity Better — captures token-level interactions (negation, qualification)
Scale Millions of documents (FAISS handles this) Top 20-50 candidates only (too slow for full corpus)
Use in P5 First-stage retrieval: embed → FAISS search → top-50 Second-stage reranking: score top-50 → return top-5

# optimizes for RECALL (find all potentially relevant docs).
# The cross-encoder optimizes for PRECISION (rank the truly

NDCG — The Ranking Metric You Haven't Used Yet

In P2, you measured Precision@K, Recall@K, and MRR. These treat relevance as binary — a result is either relevant or not. NDCG (Normalized Discounted Cumulative Gain) handles graded relevance and explicitly rewards placing more relevant results higher in the ranking. This is the standard metric for search engine evaluation.

First Principle

Not all "relevant" results are equally relevant. A chunk that directly answers the question is more relevant than a chunk that provides useful background. Precision@K treats both as "1". NDCG gives the direct answer a score of 3 and the background a score of 1, then penalizes you more for putting the "3" result at position 5 instead of position 1. The discount function (log₂(rank+1)) embodies the insight that users are less likely to look at lower-ranked results — a relevant result at position 1 is worth more than the same result at position 10.

The Math, Step by Step

DCG@K = Σᵢ₌₁ᵏ (relevance_i / log₂(i + 1))

Each position i contributes: relevance of the result at that position, divided by a logarithmic discount that increases with position. Position 1: discount = log₂(2) = 1.0. Position 5: discount = log₂(6) = 2.58. So a relevance-3 result at position 1 contributes 3.0, but at position 5 it contributes only 3/2.58 = 1.16.

NDCG@K = DCG@K / Ideal_DCG@K

Ideal DCG is the DCG you'd get if you ranked results in perfect order (most relevant first). NDCG normalizes to [0, 1] so you can compare across queries with different numbers of relevant results.

# WHY implement NDCG from scratch: Understanding the math matters more
# than calling sklearn.metrics.ndcg_score. In an interview, you need to
# explain WHY position 1 matters more than position 5.

import numpy as np

def dcg_at_k(relevances: list[float], k: int) -> float:
    """Discounted Cumulative Gain at position K."""
    relevances = np.array(relevances[:k])
    # WHY log2(i + 2): Positions are 0-indexed, but the DCG formula
    # uses 1-indexed positions with log2(position + 1).
    # Position 0 → log2(0 + 2) = log2(2) = 1.0  (no discount)
    # Position 1 → log2(1 + 2) = log2(3) = 1.58  (mild discount)
    # Position 4 → log2(4 + 2) = log2(6) = 2.58  (significant discount)
    discounts = np.log2(np.arange(len(relevances)) + 2)
    return np.sum(relevances / discounts)

def ndcg_at_k(relevances: list[float], k: int) -> float:
    """Normalized DCG — compares actual ranking to ideal ranking."""
    dcg = dcg_at_k(relevances, k)
    # WHY sort descending: The ideal ranking puts the MOST relevant
    # results first. This is the theoretical best you could achieve.
    ideal = dcg_at_k(sorted(relevances, reverse=True), k)
    if ideal == 0:
        return 0.0  # No relevant results exist → NDCG is 0
    return dcg / ideal

# Example: Your retriever returned 5 results with these relevance scores
relevances = [3, 0, 2, 0, 1]  # 3=highly relevant, 0=irrelevant
# DCG@5 = 3/1.0 + 0/1.58 + 2/2.0 + 0/2.32 + 1/2.58 = 3.0 + 0 + 1.0 + 0 + 0.39 = 4.39
# Ideal order: [3, 2, 1, 0, 0] → IDCG@5 = 3/1.0 + 2/1.58 + 1/2.0 + 0 + 0 = 4.77
# NDCG@5 = 4.39 / 4.77 = 0.92 — good ranking!

NDCG vs Precision/Recall — When Each Metric Shines

MetricRelevance ModelPosition-Sensitive?Best For
Precision@K Binary (relevant/not) No — treats all K positions equally "How many of my top results are relevant?"
Recall@K Binary No "Did I find all the relevant results?"
MRR Binary Yes — only cares about first relevant result "How quickly do I find the first good result?"
NDCG@K Graded (0, 1, 2, 3…) Yes — log discount penalizes bad ranking "Are my best results ranked highest?" — the gold standard
Pitfall

NDCG requires graded relevance judgments. If your ground truth is binary (relevant/not), NDCG still works but degenerates toward Precision@K in behavior. For the richest signal, use your LLM-as-Judge to assign relevance scores (1–5) to each retrieved chunk, not just binary labels. Your synthetic QA generation can create ground truth with graded relevance by generating "direct answer" chunks (relevance=3), "supporting context" chunks (relevance=1), and irrelevant chunks (relevance=0).

Ground Truth Generation — Building Your Evaluation Dataset

Precision, Recall, MRR, and NDCG are all meaningless without ground truth — a labeled dataset that says "for this query, these chunks are relevant." The spec offers three approaches. This section explains each and why synthetic generation is your starting point.

First Principle

You can't measure retrieval quality without knowing the right answer. This is the evaluation bootstrap problem: you need labeled data to evaluate a system, but labeling data manually is expensive. The solution is the same one you used in P1 and P4 — use an LLM to generate synthetic data with known ground truth, then validate a subset manually. The labels aren't perfect, but they're reproducible, scalable, and sufficient for comparative evaluation ("is Config A better than Config B?").

Approach 1: Synthetic QA Generation (Recommended Start)

How It Works

For each chunk in your corpus, ask an LLM to generate a question that the chunk answers. The chunk ID becomes the ground truth. At retrieval time, you check whether the system retrieves the source chunk for the generated question. This is the same pattern behind P1's synthetic data generation — you're creating controlled pairs where the answer is known by construction.

Key advantage: Scales automatically with your corpus. Ingest 100 PDFs → generate 500 QA pairs → run retrieval metrics. No manual labeling needed.

# WHY generate from chunks, not documents: The ground truth must be
# at CHUNK granularity because that's what the retriever returns.
# A document-level question like "What is this paper about?" has
# ALL chunks as ground truth — useless for measuring precision.

from pydantic import BaseModel

class SyntheticQAPair(BaseModel):
    question: str
    source_chunk_id: str        # Ground truth — the chunk that answers this question
    source_document: str        # For citation verification
    difficulty: str = "medium"  # "easy" (verbatim answer in chunk), "hard" (requires inference)

GENERATE_QA_PROMPT = """Given this text chunk from a document, generate a specific
factual question that can ONLY be answered using information in this chunk.

Chunk (from {source}, page {page}):
{chunk_text}

Requirements:
- Question must be answerable from this chunk alone
- Question should be specific (not "What is discussed?")
- Question should sound like a real user query

Return JSON: {{"question": "...", "difficulty": "easy|medium|hard"}}"""

async def generate_qa_pairs(chunks: list[Chunk], llm: BaseLLM, n_per_chunk: int = 1) -> list[SyntheticQAPair]:
    """Generate synthetic QA pairs with known ground truth chunk IDs."""
    pairs = []
    for chunk in chunks:
        prompt = GENERATE_QA_PROMPT.format(
            source=chunk.metadata.source,
            page=chunk.metadata.page_number,
            chunk_text=chunk.content
        )
        # WHY Instructor/Pydantic here: Same pattern as P1/P4 —
        # structured output guarantees the LLM returns parseable JSON
        response = await llm.generate(prompt)
        pairs.append(SyntheticQAPair(
            question=response.question,
            source_chunk_id=chunk.id,
            source_document=chunk.metadata.source,
            difficulty=response.difficulty
        ))
    return pairs

Approach 2: Manual Labeling (Gold Standard Subset)

How It Works

Write 10–20 queries by hand. For each query, manually identify every relevant chunk in the corpus. Store as {query, relevant_chunk_ids} pairs. This is labor-intensive but produces the highest-quality ground truth. Use this for your "critical queries" — the ones where you need absolute confidence in the metrics.

When to use: After synthetic generation gives you broad coverage. Pick 10 queries where synthetic and manual labels disagree — those disagreements reveal chunking problems or question ambiguity.

Approach 3: LLM-as-Judge Only (No Retrieval Metrics)

How It Works

Skip ground truth entirely. Run queries through the full pipeline (retrieve → generate), then use an LLM judge to score the generated answer on relevance, accuracy, completeness, and citation quality. You can't compute Precision/Recall/MRR/NDCG this way (those require labeled retrieval ground truth), but you can still evaluate end-to-end quality.

Trade-off: Faster, but you lose the ability to diagnose where the pipeline fails. If the answer is bad, was it retrieval or generation? Without retrieval metrics, you can't tell. This is why P2's retrieval-generation gap was such a key finding — you needed both metrics to see it.

Graded Relevance for NDCG

Key Insight

NDCG needs graded relevance, not binary. Use your LLM to assign relevance grades when generating synthetic QA: the source chunk gets relevance=3 (directly answers the question), chunks from the same page or section get relevance=1 (supporting context), everything else gets relevance=0 (irrelevant). This three-tier scheme gives NDCG meaningful signal — it can distinguish a retriever that puts the exact answer at position 1 from one that puts a "related but not direct" chunk at position 1.

Pitfall

Don't trust synthetic QA blindly. In P1, you discovered that AI-generated data requires systematic validation — the same applies here. Spot-check 10–15% of your generated QA pairs: Is the question actually answerable from the chunk? Is it specific enough that only 1–3 chunks are truly relevant? If the LLM generates vague questions like "What are the key points?", your ground truth will have inflated recall because many chunks match.

Java/TypeScript Parallel

This is the same concept as generating test fixtures in JUnit or Jest. In Java, you'd use a @BeforeAll method to populate Map<String, Set<String>> groundTruth. The synthetic approach is like using a TestDataFactory that auto-generates test inputs with known expected outputs. The manual approach is hand-written @ParameterizedTest cases. Production codebases use both — auto-generated for coverage, hand-written for critical paths.

LLM-as-Judge for Generation Quality — The 4-Axis Rubric

You've used LLM-as-Judge in P1 (evaluating synthetic data quality) and P2 (via RAGAS). P5 requires a custom judge that evaluates generated answers on four specific axes: Relevance, Accuracy, Completeness, and Citation Quality. The target is an average score >4.0 across all criteria on a 1–5 scale.

First Principle

Retrieval metrics (Precision, Recall, MRR, NDCG) measure the input to the LLM. Generation quality metrics measure the output. Both are necessary because they fail independently — P2 proved this with 0.747 Recall@5 but only 0.511 faithfulness. A system can retrieve perfectly but generate poorly (hallucination, ignoring context) or retrieve poorly but generate well on what little context it has. P5's four-axis rubric gives you diagnostic power: if Relevance is high but Citation Quality is low, the LLM understands the question but isn't grounding its answer.

The Four Axes

Relevance (1–5): Does the answer address the user's actual question? A score of 1 means the answer is about a completely different topic. A score of 5 means it directly and specifically answers what was asked.

Accuracy (1–5): Is the information factually correct based on the source documents? A score of 1 means the answer contains fabricated or contradictory information. A score of 5 means every claim is verifiable in the provided context.

Completeness (1–5): Is the answer thorough enough? A score of 1 means it addresses only a fragment of the question. A score of 5 means it covers all aspects the source documents can support.

Citation Quality (1–5): Are sources properly attributed? A score of 1 means no citations or entirely wrong ones. A score of 5 means every substantive claim has a correct, traceable citation.

# WHY a structured rubric prompt: Vague instructions like "rate quality 1-5"
# produce inconsistent scores. Anchored rubrics with level descriptors
# constrain the judge. This is the same principle behind Cohen's Kappa
# in P1 — inter-rater agreement improves when the rating scale is explicit.

JUDGE_PROMPT = """You are evaluating a RAG system's generated answer.

QUESTION: {question}

RETRIEVED CONTEXT:
{context_chunks}

GENERATED ANSWER:
{answer}

Rate each dimension on a 1-5 scale using these anchors:

RELEVANCE (Does the answer address the question?)
1: Completely off-topic  2: Tangentially related  3: Partially addresses question
4: Mostly addresses question  5: Directly and specifically answers

ACCURACY (Is information factually correct per the source context?)
1: Major factual errors  2: Several inaccuracies  3: Minor errors present
4: Mostly accurate  5: Every claim verifiable in context

COMPLETENESS (Is the answer thorough?)
1: Addresses fragment only  2: Missing major aspects  3: Covers basics
4: Covers most aspects  5: Comprehensive within source limits

CITATION QUALITY (Are sources properly attributed?)
1: No citations or all wrong  2: Few/mostly wrong citations  3: Some correct citations
4: Most claims cited correctly  5: Every substantive claim properly cited

Return JSON only:
{{"relevance": N, "accuracy": N, "completeness": N, "citation_quality": N, "reasoning": "..."}}"""

class GenerationJudgment(BaseModel):
    relevance: int = Field(ge=1, le=5)
    accuracy: int = Field(ge=1, le=5)
    completeness: int = Field(ge=1, le=5)
    citation_quality: int = Field(ge=1, le=5)
    reasoning: str

    @property
    def average_score(self) -> float:
        # WHY average of all 4: The spec targets >4.0 average.
        # Tracking each axis separately reveals WHERE quality drops.
        return (self.relevance + self.accuracy + self.completeness + self.citation_quality) / 4.0

Aggregation Strategy

Per-Query and Per-Axis Aggregation

Run the judge on every (query, answer) pair in your test set (10+ queries per experiment). Aggregate two ways: (1) per-query average — the mean across all 4 axes for a single query, to find queries where the system struggles, and (2) per-axis average — the mean across all queries for a single axis, to find systematic weaknesses. If Citation Quality averages 2.8 while Relevance averages 4.3, you know the generation prompt needs better citation instructions, not better retrieval.

Key Insight

The LLM judge sees the retrieved context, not just the answer. This is critical — it can detect hallucination (answer claims something not in context → Accuracy drops) and missed context (answer ignores relevant chunks → Completeness drops). This is a more granular version of what RAGAS measured in P2, but with four axes instead of two, and you control the rubric directly instead of relying on RAGAS's opaque scoring.

Pitfall from P1

LLM judges are biased toward leniency — they tend to rate 4–5 even when quality is mediocre. In P1, you addressed this by calibrating against manual spot-checks. Do the same here: manually score 5 answers yourself, then compare to the judge's scores. If the judge consistently rates 1 point higher than you, adjust your threshold (e.g., treat judge-4.0 as "acceptable" instead of the spec's >4.0).

🎯 Analogy — Restaurant Review

Imagine reviewing a restaurant on four axes: Taste (Relevance — did the food match what you ordered?), Ingredients (Accuracy — is it actually wagyu beef or just regular beef?), Portion Size (Completeness — enough food to satisfy?), and Menu Description (Citation Quality — did the menu accurately describe what arrived?). A restaurant scoring 5/5 on Taste but 1/5 on Menu Description is a great chef with a lying menu — that's your RAG system generating good answers but with wrong citations. Each axis tells you a different thing to fix.

Citation Extraction — Grounding LLM Output

An answer without a citation is an assertion without evidence. P5 requires that every generated answer includes traceable references to the source chunks, so users can verify claims against the original document. This is what separates a production QA system from a "vibes-based" chatbot.

First Principle

LLMs are probabilistic text generators, not truth engines. Even when given perfect context, they can hallucinate, misattribute, or subtly distort information. Citations create a verification contract: the user can check whether the generated answer actually matches the source material. This is the same principle behind academic citations and audit trails — traceability enables trust.

Two Citation Approaches

Approach 1: Structured Format

Prompt the LLM to cite using [Source: filename.pdf, Page: 12] format. Parse with regex after generation. Simple and reliable for well-structured prompts.

Approach 2: Index References (Recommended)

Number each context chunk in the prompt: "[1] chunk_text_1 [2] chunk_text_2…". The LLM cites using [1], [2], etc. Parse the numbers, map back to chunk objects. More robust because the LLM only needs to output a number, not reconstruct filenames.

# WHY index references over structured format: The LLM is more reliable
# at producing "[1]" than "[Source: technical-manual-v3.2.pdf, Page: 47]".
# Fewer tokens to generate = fewer chances for hallucination.

PROMPT_TEMPLATE = """Answer the question using ONLY the provided context.
Cite your sources using [N] notation.

Context:
{numbered_chunks}

Question: {question}

Answer (cite with [1], [2], etc.):"""

def build_prompt(question: str, chunks: list[Chunk]) -> str:
    numbered = "\n\n".join(
        f"[{i+1}] {c.content}" for i, c in enumerate(chunks)
    )
    return PROMPT_TEMPLATE.format(numbered_chunks=numbered, question=question)

import re
def extract_citations(answer: str, chunks: list[Chunk]) -> list[Citation]:
    """Parse [N] references from LLM answer, map to chunk metadata."""
    # WHY regex r'\[(\d+)\]': Matches [1], [2], etc. in the answer text.
    # Each match maps to the chunk at that index (1-based).
    refs = set(re.findall(r'\[(\d+)\]', answer))
    citations = []
    for ref in refs:
        idx = int(ref) - 1  # Convert 1-based to 0-based
        if 0 <= idx < len(chunks):
            chunk = chunks[idx]
            citations.append(Citation(
                chunk_id=chunk.id,
                source=chunk.metadata.source,
                page_number=chunk.metadata.page_number,
                text_snippet=chunk.content[:200]
            ))
    return citations

Abstract Base Classes — The Strategy Pattern in Python

P5 requires that all components implement abstract interfaces: BaseChunker, BaseEmbedder, BaseVectorStore, BaseRetriever, BaseReranker, BaseLLM. This is the Strategy Pattern — the same OOP pattern you know from Java/TypeScript interfaces. In Python, this is done with the abc module.

Java/TypeScript Parallel

In Java: interface Retriever { List<Result> retrieve(String query, int topK); } and class DenseRetriever implements Retriever, class BM25Retriever implements Retriever. In TypeScript: interface Retriever { retrieve(query: string, topK: number): Result[]; }. Python's ABC is the direct equivalent — @abstractmethod is like Java's abstract keyword. The critical difference: Python doesn't enforce interface contracts at compile time. The error only surfaces when you try to instantiate a class that hasn't implemented all abstract methods. This is a runtime check, not a compile-time check.

# WHY ABC: Python doesn't have interfaces like Java/TypeScript.
# ABC (Abstract Base Class) is the closest equivalent. It enforces
# that subclasses implement required methods — but at RUNTIME, not compile time.
# If you forget to implement an @abstractmethod, you get a TypeError
# when you try to instantiate the class, not when you define it.

from abc import ABC, abstractmethod
from typing import Protocol  # Alternative: structural subtyping

    @abstractmethod
    def chunk(self, document: Document) -> list[Chunk]:
        """Split a document into chunks. Every chunker must implement this."""
        ...

class RecursiveChunker(BaseChunker):
    def __init__(self, chunk_size: int = 512, overlap: int = 50):
        self.chunk_size = chunk_size
        self.overlap = overlap

    def chunk(self, document: Document) -> list[Chunk]:
        # Concrete implementation
        ...

# WHY Protocol as an alternative: Python 3.8+ supports "structural subtyping"
# via Protocol. This is like TypeScript's duck typing — if an object has
# the right methods, it satisfies the Protocol, without explicit inheritance.
# ABC = nominal typing (Java-style).  Protocol = structural typing (TS-style).
# For P5, ABC is clearer because it explicitly declares the contract.

The real power of this pattern isn't just code organization — it's configuration-driven experimentation. This is the Strategy Pattern combined with the Factory Pattern — and it's exactly how production ML systems are built.

PDF Processing — PyMuPDF and Text Extraction

P2 and P3 worked with pre-cleaned text. P5 starts from raw PDFs, which are messy: multi-column layouts, headers/footers on every page, tables rendered as text, ligatures, and encoding artifacts. PyMuPDF (imported as fitz) is the extraction library — fast, pure-Python, and handles most PDF quirks.

# WHY PyMuPDF (fitz) over alternatives:
# - pdfplumber: Better for tables, but slower
# - PyPDF2: Simpler API, but worse text extraction quality
# - PyMuPDF: Best balance of speed + quality for general text extraction
# The library name is 'fitz' for historical reasons (MuPDF's founder)

import fitz  # pip install PyMuPDF

def extract_text_from_pdf(pdf_path: str) -> Document:
    """Extract text and metadata from a PDF file."""
    doc = fitz.open(pdf_path)
    pages = []
    for page_num, page in enumerate(doc):
        text = page.get_text("text")  # "text" = plain text extraction
        # WHY clean the text: PDFs embed formatting artifacts:
        # - Repeated headers/footers on every page
        # - Hyphenated line breaks ("infor-\nmation" → "information")
        # - Multiple spaces from column alignment
        # - Non-breaking spaces, ligatures (ﬁ, ﬂ)
        text = clean_pdf_text(text)
        pages.append({"page_number": page_num + 1, "content": text})

    # WHY extract metadata: Title, author, creation date help with
    # citation generation and document-level filtering
    metadata = doc.metadata  # dict with title, author, subject, etc.
    return Document(
        content="\n\n".join(p["content"] for p in pages),
        metadata={"source": pdf_path, **metadata, "page_count": len(doc)},
        pages=pages
    )
Pitfall

PDFs are not text documents. They're layout documents that happen to contain text. A two-column academic paper will extract as interleaved text from both columns unless you handle the layout. Headers/footers repeat on every page — if you don't strip them, your chunks will be full of "Page 12 | Technical Manual v3.2" noise. Always inspect your raw extracted text before chunking.

Java/TypeScript Parallel

In Java, you'd use Apache PDFBox (PDDocument → PDFTextStripper). Same concept, same gotchas. The Python fitz.open() is like PDDocument.load() — it returns a document object you iterate over page by page. The with context manager in Python handles resource cleanup like Java's try-with-resources.

Loads a saved index from disk and enters a REPL loop: accepts questions, retrieves chunks, generates answers with citations, prints results. This is the "quick test" interface for development — faster to iterate than launching Streamlit.

# Main area — question and answer
question = st.text_input("🔍 Ask a question about your documents")

if question and st.session_state.index:
    response = pipeline.query(question, top_k=top_k)
    st.markdown(f"### Answer\n{response.answer}")

    # Source chunks with expandable detail
    st.markdown("### 📚 Sources")
    for citation in response.citations:
        with st.expander(f"📄 {citation.source} — Page {citation.page_number}"):
            st.write(citation.text_snippet)
            st.caption(f"Relevance: {citation.relevance_score:.2f}")
Java/TypeScript Parallel

Think of Streamlit's rerun model as a server-side React with no virtual DOM diff. Every user interaction triggers a full re-render of the Python script. st.session_state is your equivalent of useState — it survives reruns. The sidebar is like a configuration panel in a Spring Boot admin UI. The key mental shift from React: there's no component lifecycle. The script runs top-to-bottom every time. State lives in session_state, not in component instances.
